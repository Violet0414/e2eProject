# 测试点评审

对测试点清单进行系统评审，检查覆盖率、一致性，输出评审报告。

## 输入

- 测试点清单（通常由"测试点提取"技能产出）
- 可选：原始功能点列表，用于交叉比对覆盖率
- 可选：需求文档，用于验证测试点与需求的对齐
- **必读参考文件**：`./referenceDocument/testPointsRequirement.md`，覆盖率检查时必须对照此文件中的检查项

## 评审维度

### 1. 覆盖率检查
- 每个功能点是否都有对应的测试点？
- 每个功能点的正向/异常/边界场景是否覆盖？
- 是否存在功能点被遗漏（无任何测试点）？
- 需求文档中明确提到的规则/约束是否有对应测试点？
- **对照 referenceDocument/testPointsRequirement.md**：根据每个功能点的"功能类型"，逐项核对需求文件中对应类别的检查项是否都有测试点覆盖，遗漏的检查项列入问题清单

### 2. 一致性检查
- 测试点描述是否与需求/功能点语义一致？
- 不同测试点之间是否存在矛盾？
- 分类标注是否准确（正向标为异常等）？

### 3. 质量检查
- 测试点粒度是否合规（一个场景一句话）？
- 测试点是否可验证（描述是否足够明确）？
- 是否存在重复的测试点？

### 4. 完整性检查
- 是否遗漏常见场景（如：空值、超长输入、并发、网络异常）？
- 多角色/多权限场景是否覆盖？
- 关联功能之间的交互场景是否覆盖？

## 输出格式

```markdown
## 测试点评审报告

**评审日期**：{当前日期}
**评审结论**：✅ 通过 / ⚠️ 需修改

### 评审统计

| 维度 | 检查项数 | 通过 | 需修改 |
|------|---------|------|--------|
| 覆盖率 |  |  |  |
| 一致性 |  |  |  |
| 质量   |  |  |  |
| 完整性 |  |  |  |

### 问题清单

| 编号 | 关联测试点 | 维度 | 问题描述 | 建议修改 |
|------|-----------|------|---------|---------|
| R01  |           |      |         |         |

### 遗漏建议

- 建议补充的测试点：{列出建议新增的测试点}
```

## 注意事项

- 评审结论为"需修改"时，必须给出具体修改建议
- 如果未提供功能点列表或需求文档，仅基于测试点本身进行内部一致性检查，并在报告中注明评审范围受限
- **回退机制**：评审结论为"需修改"时，在问题清单中明确列出需修改/新增的测试点编号及建议内容；用户可重新触发"测试点提取"并指定需修正的功能点范围，无需重新提取全部测试点

## 持久化

- 评审报告必须同时写入文件 `./output/{当前日期}/测试点评审报告.md`（如目录不存在则先创建，"当前日期"格式为YYYY-MM-DD，如2026-05-15）
- 文件已存在时询问用户是否覆盖
