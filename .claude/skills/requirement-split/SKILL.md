# 需求拆分

将输入的需求文档或PRD拆分为结构化的功能点列表。

## 输入

- 需求文档/PRD（文本、文件路径或用户粘贴的内容）

## 处理规则

1. **通读全文**，理解需求背景、目标和范围
2. **按用户可感知的功能维度拆分**，每个功能点应是一个用户可以直接体验/操作的单元
3. 拆分粒度要求：
   - 太粗：一个功能点包含多个独立操作 → 继续拆
   - 太细：一个操作步骤被单独列为功能点 → 合并到所属功能
   - 刚好：用户能感知"我做了这件事，得到了这个结果"
4. 排除纯技术实现细节（如数据库字段、接口设计），除非它们对用户有直接影响
5. **功能类型标注**：为每个功能点标注类型，取值：菜单 / 列表查询 / 列表数据 / 列表翻页 / 列表操作 / 新增表单 / 编辑表单 / 详情表单 / 其他
6. **接口监测例外**：对于涉及接口交互的功能点，即使属于技术层面，仍保留"接口传参与响应校验"作为独立功能点

## 输出格式

```markdown
## 功能点列表

**需求名称**：{PRD标题}
**拆分日期**：{当前日期}

| 编号 | 模块 | 功能点 | 描述 | 功能类型 | 优先级 | 备注 |
|------|------|--------|------|---------|--------|------|
| F01  |      |        |      | 菜单/列表查询/列表数据/列表翻页/列表操作/新增表单/编辑表单/详情表单/其他 | P0/P1/P2 |    |
| F02  |      |        |      |         |        |    |
```

## 注意事项

- 如果输入内容模糊或缺少关键信息，主动向用户确认
- 功能点之间应尽量正交，减少交叉依赖
- 优先级基于需求文档中的表述；若文档未明确，根据功能核心程度推断并标注

## 持久化

- 生成结果必须同时写入文件 `./output/{当前日期}/功能点列表.md`（如目录不存在则先创建，"当前日期"格式为YYYY-MM-DD，如2026-05-15）
- 文件已存在时询问用户是否覆盖
