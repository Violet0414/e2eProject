# 测试点提取

从功能点列表中提取测试点清单，每个测试点对应一个可验证的场景。

## 输入

- 功能点列表（结构化表格，通常由"需求拆分"技能产出）
- 可选：补充的业务规则或约束说明
- **必读参考文件**：`./referenceDocument/testPointsRequirement.md`，提取测试点时必须参照此文件中的检查要求

## 处理规则

1. **读取参考文件**：先读取 `./referenceDocument/testPointsRequirement.md`，理解其中定义的各类检查要求（菜单、列表查询、列表数据、列表翻页、列表操作、新增表单、编辑表单、详情表单、接口监测等）
2. **逐个功能点分析**，针对每个功能点提取所有需要验证的场景
3. **参照参考文件匹配检查项**：根据功能点的类型（如菜单、列表、表单、详情等），从 `referenceDocument/testPointsRequirement.md` 中找到对应的检查类别，将其中适用的检查要求转化为测试点
4. **粒度要求**：一个测试点 = 一个验证场景 = 一句话描述
   - 正确示例："用户使用正确账号密码登录成功"
   - 错误示例："测试登录功能"（太粗）
   - 错误示例："用户在登录页面输入账号后点击登录按钮系统校验通过跳转首页"（太细/多步骤）
5. **覆盖维度**（每个功能点至少考虑）：
   - 正向场景：正常流程、主路径
   - 异常场景：输入异常、操作中断、边界值
   - 权限/角色：不同角色看到的差异
   - 数据状态：空数据、大数据量、特殊字符
   - 兼容/环境：如需求涉及则补充
6. 不需要写成完整用例，只提炼"验证什么"
7. **优先级继承**：测试点的优先级继承自关联功能点的优先级；若功能点有多个优先级不同子场景，按场景核心程度调整

## 输出格式

```markdown
## 测试点清单

**关联需求**：{需求名称}
**提取日期**：{当前日期}

| 编号 | 关联功能点 | 功能类型 | 测试点 | 分类 | 优先级 | 备注 |
|------|-----------|---------|--------|------|--------|------|
| TP01 | F01       |         |        | 正向/异常/边界/权限/数据 | P0/P1/P2 |   |
| TP02 | F01       |         |        |      |        |
```

## 注意事项

- 测试点之间不应有冗余重复
- 边界值场景需明确指出边界条件
- 如果功能点描述不够清晰无法提取测试点，列出疑问向用户确认

## 持久化

- 生成结果必须同时写入文件 `./output/{当前日期}/测试点清单.md`（如目录不存在则先创建，"当前日期"格式为YYYY-MM-DD，如2026-05-15）
- 文件已存在时询问用户是否覆盖
